## 4. Task A — Opus 重跑 Phase II 迭代分析

### 4.1 输入定义


| 字段     | 值                                                                                                                                                                  |
| ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 目标变异体  | 365 个，来源：`stress_enhance_results/details/*.json` 中 `any_killed == false`                                                                                           |
| 携带证据   | Phase I `equiv_detail`（L0-L3） + Phase II `main_track` (`value_stress / dtype_stress / repeated_run / training_stress`) + Phase II `config_track` (`config_stress`) |
| Prompt | **完全复用**现有 `build_analysis_prompt` / `build_reanalysis_prompt`（一字不改），见 [提示词全集 §1-2](提示词全集.md#task-a--analysis_prompt-v2)                                           |


### 4.2 单 mutant 处理流程

```
对每个 mutant_id (共 365 个):
    1. 加载 Phase II detail (equiv_detail, main_track, config_track, baseline)
    2. 加载 Phase I detail 的 mutated_code 和 original_code (完整源码)
    3. 解析 problem_file 路径（KernelBench level{N}/{id}_*.py）

    rounds_history = []
    for round_num in 1..5:
        if round_num == 1:
            prompt = build_analysis_prompt(...)           # ANALYSIS_PROMPT V2
        else:
            prompt = build_reanalysis_prompt(             # REANALYSIS_PROMPT V2
                ..., previous_rounds=rounds_history)

        保存 prompts/{mutant_id}_r{round_num}.txt          ← 完整 prompt 落盘

        llm_resp = call_opus_45(prompt)                    # extended_thinking=ON
        保存 llm_responses/{mutant_id}_r{round_num}_response.json
            # 含 content + reasoning_content + usage + latency_ms

        parsed = parse_llm_response(llm_resp.content)      # 提取 JSON

        if parsed is None:                                 # JSON 解析失败
            记录 parse_error 到 rounds_history; continue

        if parsed.killable == False:                       # LLM 判定不可杀
            记录到 rounds_history; break                    ← 早停

        python_code = parsed.suggested_test.python_code
        if 违反 fixed-shape 关键词:
            记录 violates_fixed_shape; continue            ← 跳过本轮

        exec_result = _verify_llm_suggestion(              # 子进程跑 CUDA
            problem_file, original_code, mutated_code, python_code)
        # exec_result 含 killed / ref_ok / original_ok / mutant_ok / diff_summary / error

        记录 round_record 到 rounds_history

        if exec_result.killed:
            killed = True; killing_round = round_num; break

    保存 details/{mutant_id}.json (整合所有轮次 + DeepSeek baseline 对比)
    更新 completed.json
    更新 run_manifest.json (累计 token / 杀死数)
```

### 4.3 早停规则


| 条件                         | 行为                                |
| -------------------------- | --------------------------------- |
| LLM 判定 `killable=false`    | **整体 break**（不再迭代）                |
| LLM `suggested_test=null`  | 同上 break                          |
| Python 代码违反固定 shape（关键词检测） | 跳过本轮、`continue` 下一轮               |
| 解析 JSON 失败                 | 记录 `parse_error`、`continue` 下一轮   |
| 子进程超时/崩溃                   | 记录 `timeout/crash`、`continue` 下一轮 |
| 子进程返回 `killed=true`        | 标记 killed、**整体 break**            |
| 走完第 5 轮仍未杀死                | killed=false、自然结束                 |

## 6. Task B — Opus 修复在增强测试中失败的 kernel

> **状态**：脚本待实现；前置数据采集（train mode 补跑 + diff_summary 抓取）在 Task C 完成后启动，随后立即进入 Task B 主流程。

### 6.1 任务目标的精确定义

**Task B 目标 kernel = 同时满足两个条件**：

| 条件 | 定义 | 数据来源 |
|---|---|---|
| **① 原始测试通过** | 在 KernelBench 默认 `get_inputs()` 生成的随机输入下，optimized kernel 与参考实现的输出满足 `torch.allclose`（KernelBench 原生验证），即 `num_correct_turns ≥ 1` | `best_kernels.json`（90 个 kernel） |
| **② 增强测试失败** | 在 Phase II 增强测试某维度下，**存在某具体输入 `s` 使得**：参考实现跑得通（`ref_ok=True`） 且 原 kernel 跑不通（`original_ok=False`） | `stress_enhance_results/details/*.json` 中 `main_track.{value_stress, repeated_run, training_stress}.policy_results` |

> **关键术语澄清**：
> - **"参考实现" = KernelBench 提供的 PyTorch 代码**（即 `level{N}/{id}_*.py` 中的 `class Model(nn.Module)`），用 `torch.nn` 算子直接搭建，跑在 GPU 上得到 ground-truth 输出。
> - **"原 kernel"** = `best_kernels.json` 收录的 optimized 实现（`ModelNew`，通常含 CUDA `load_inline` 或 Triton）。
> - **"增强测试"** = Phase II 的全部 stress 维度，**不限于 mutation testing 本身**——这里关心的是 *输入压力* 对 kernel 自身的影响。

#### 6.2.3 前置补跑 A：train mode 的 `ref_ok` 字段

`main_track.training_stress.results` 因原代码未保存 `ref_ok`，无法严格判定 train mode 候选是否符合"参考实现跑得通"。

**Task C 跑完后立刻执行的补跑**（约 30 分钟）：

```text
脚本：scripts/_supplement_train_refok.py（新建）

  对 18 个 buggy kernel 中"有 train 候选"的 12 个 kernel，
  对每个 train 候选 (policy, seed):
      复用 _stress_worker.py 的 run_training_stress() 子进程逻辑，
      只关心 ref_ok 真值，写入新文件
      task_b_regenerate/_train_refok_supplement.json

  过滤规则：保留 ref_ok=True 且 original_ok=False 的 case
  并入 Task B 的 failing inputs 总池
```

#### 6.2.4 前置补跑 B：单次失败输入的 `diff_summary` 抓取

eval 数据虽记录了 `ref_ok / original_ok / error`，但**没存** `ref_range / orig_range / max_diff`（这些信息在 `_stress_worker.py` 中算了，但 `run_stress_enhance.py` 未保存到 detail JSON）。

Task B prompt 需要这些数值才有意义，所以**在每个 buggy kernel 进入 Task B 前**：

```text
对该 kernel 的所有失败 (policy, seed, mode):
    重新调用 _stress_worker 的 run_stress / run_training_stress
    抓取完整 diff_summary（max_diff / mean_diff / ref_range / orig_range）
    缓存到 task_b_regenerate/failing_inputs_cache/{kernel_name}.json
```

开销估算：18 kernel × 平均 50 个 case × ~3s = **~45 分钟**，与 Task B 主流程串行进行。

### 6.3 单 kernel 处理流程（3 轮迭代）

```
对每个 buggy kernel（共 18 个）:

  # 阶段 0：失败输入准备
  1. 读 task_b_buggy_kernels_from_existing_data.json 获得该 kernel 的失败清单
  2. 合并 _train_refok_supplement.json 中的 train mode 失败（若有）
  3. 调 _stress_worker 抓取每个失败输入的 diff_summary
     失败清单 failing_inputs = [{policy, seed, mode, ref_range, orig_range,
                                 max_diff, mean_diff, error, ...}, ...]

  # 阶段 1：3 轮迭代修复
  iut ← 原 buggy kernel 源代码
  last_round_failures_stress ← []
  last_round_failures_kb     ← []

  for round_num in 1..3:
      if round_num == 1:
          prompt = build_taskB_initial_prompt(
              reference_code,  # PyTorch Model 源码
              iut,             # 当前 kernel
              failing_inputs)
      else:
          prompt = build_taskB_iterate_prompt(
              reference_code, iut, failing_inputs,
              last_round_failures_stress,
              last_round_failures_kb)

      保存 prompts/{kernel_name}_r{round_num}.txt
      llm_resp = call_opus_45(prompt, thinking=ON, budget=8000)
      保存 llm_responses/{kernel_name}_r{round_num}_response.json

      new_kernel = extract_python_block(llm_resp.content)

      # Syntax 自修（最多 2 子轮）
      compile_ok, err = try_compile(new_kernel)
      if not compile_ok:
          for syn_retry in 1..2:
              syn_prompt = "上次编译失败:\n{err}\n请修复并重新输出完整 kernel"
              new_kernel = LLM(syn_prompt + new_kernel)
              compile_ok, err = try_compile(new_kernel)
              if compile_ok: break
      if not compile_ok:
          记录 round_record(status="compile_failed"); continue

      # V_stress：失败输入集验证
      v_stress = verify_failing_inputs(
          new_kernel, failing_inputs, atol=1e-2, rtol=1e-2)

      # V_kb：KernelBench 默认输入验证（防 regression）
      v_kb = verify_kb_default(
          new_kernel, problem_file, n_seeds=5, atol=1e-3, rtol=1e-3)

      记录 round_record(round_num, v_stress, v_kb, kernel_path, tokens)

      if v_stress.pass == v_stress.total and v_kb.pass == v_kb.total:
          final_status = f"fixed_at_round_{round_num}"; break

      iut ← new_kernel
      last_round_failures_stress ← v_stress.still_failing
      last_round_failures_kb     ← v_kb.regressed_seeds

  # 阶段 2：写最终输出
  if 未在 3 轮内成功:
      final_status = "failed_after_3_rounds"
  保存 details/{kernel_name}.json + kernels/{kernel_name}_round{N}.py
```

### 6.5 双重验证 V_stress + V_kb 的精确定义

| 验证 | 输入来源 | 容差 | 通过条件 |
|---|---|---|---|
| **V_stress** | 该 kernel 的失败输入集（§6.2：eval + 补跑后 train） | `atol=1e-2, rtol=1e-2`（与 Phase II 一致） | **每个失败输入都满足 `torch.allclose(ref, new)` 且 new 无 NaN/Inf** |
| **V_kb** | KernelBench `get_inputs()` × 5 种子（42, 1337, 7, 100, 2024） | `atol=1e-3, rtol=1e-3`（KernelBench 默认） | **5 个种子下输出与参考实现全过** |

> 两个都过才算"修复成功"。V_stress 保证目标失败 case 被修；V_kb 兜底防止 LLM 过拟合到 failing inputs 而把正常 case 搞坏。

> **注意**：本次 Task B 不做"21 policy × 2 seed 全套增强测试"作为验证。原因是：
> - 我们要回答的是"对暴露出的失败输入能否修"，而不是"是否对所有 stress 策略都鲁棒"
> - 若 V_stress 中给定的 case 修对了但其他 policy 上又新出现失败，属于 Task A/C 的范畴
> - 简化验证逻辑、缩短迭代周期
