## 4. Task A — Opus 重跑 Phase II 迭代分析

### 4.1 输入定义

| 字段 | 值 |
|------|----|
| 目标变异体 | 365 个，来源：`stress_enhance_results/details/*.json` 中 `any_killed == false` |
| 携带证据 | Phase I `equiv_detail`（L0-L3） + Phase II `main_track` (`value_stress / dtype_stress / repeated_run / training_stress`) + Phase II `config_track` (`config_stress`) |
| Prompt | **完全复用**现有 `build_analysis_prompt` / `build_reanalysis_prompt`（一字不改），见 [提示词全集 §1-2](提示词全集.md#task-a--analysis_prompt-v2) |

### 4.2 单 mutant 处理流程

```
对每个 mutant_id (共 365 个):
    1. 加载 Phase II detail (equiv_detail, main_track, config_track, baseline)
    2. 加载 Phase I detail 的 mutated_code 和 original_code (完整源码)
    3. 解析 problem_file 路径（KernelBench level{N}/{id}_*.py）

    rounds_history = []
    for round_num in 1..5:
        if round_num == 1:
            prompt = build_analysis_prompt(...)           # ANALYSIS_PROMPT V2
        else:
            prompt = build_reanalysis_prompt(             # REANALYSIS_PROMPT V2
                ..., previous_rounds=rounds_history)

        保存 prompts/{mutant_id}_r{round_num}.txt          ← 完整 prompt 落盘

        llm_resp = call_opus_45(prompt)                    # extended_thinking=ON
        保存 llm_responses/{mutant_id}_r{round_num}_response.json
            # 含 content + reasoning_content + usage + latency_ms

        parsed = parse_llm_response(llm_resp.content)      # 提取 JSON

        if parsed is None:                                 # JSON 解析失败
            记录 parse_error 到 rounds_history; continue

        if parsed.killable == False:                       # LLM 判定不可杀
            记录到 rounds_history; break                    ← 早停

        python_code = parsed.suggested_test.python_code
        if 违反 fixed-shape 关键词:
            记录 violates_fixed_shape; continue            ← 跳过本轮

        exec_result = _verify_llm_suggestion(              # 子进程跑 CUDA
            problem_file, original_code, mutated_code, python_code)
        # exec_result 含 killed / ref_ok / original_ok / mutant_ok / diff_summary / error

        记录 round_record 到 rounds_history

        if exec_result.killed:
            killed = True; killing_round = round_num; break

    保存 details/{mutant_id}.json (整合所有轮次 + DeepSeek baseline 对比)
    更新 completed.json
    更新 run_manifest.json (累计 token / 杀死数)
```

### 4.3 早停规则

| 条件 | 行为 |
|------|------|
| LLM 判定 `killable=false` | **整体 break**（不再迭代） |
| LLM `suggested_test=null` | 同上 break |
| Python 代码违反固定 shape（关键词检测） | 跳过本轮、`continue` 下一轮 |
| 解析 JSON 失败 | 记录 `parse_error`、`continue` 下一轮 |
| 子进程超时/崩溃 | 记录 `timeout/crash`、`continue` 下一轮 |
| 子进程返回 `killed=true` | 标记 killed、**整体 break** |
| 走完第 5 轮仍未杀死 | killed=false、自然结束 |

### 4.4 输出 schema（per mutant）

```text
task_a_phase2_rerun/
├── prompts/
│   └── {mutant_id}_r{N}.txt              # 完整 prompt 原文（每轮 1 个）
├── llm_responses/
│   └── {mutant_id}_r{N}_response.json    # LLM 原始响应（含 thinking）
├── details/
│   └── {mutant_id}.json                  # 整合输出（见下）
├── completed.json                        # 已完成 mutant_id 列表（断点续跑）
├── run_manifest.json                     # 全局元数据
└── run.log                               # nohup stdout
```

       *** 不加载 Phase II ***

    for round_num in 1..5:
        if round_num == 1:

        else:
            prompt = build_phase1_direct_reanalysis_prompt(       # REANALYSIS_PROMPT_C

### 5.3 输出 schema

### 5.4 学术价值

任务完成后能回答：

| 实验结果 | 学术结论 |
|---------|---------|
| `Task C kill rate ≈ Task A kill rate` | Phase II 的 5 维确定性增强**可被强 LLM 替代**，简化 pipeline 有依据 |
| `Task C kill rate ≪ Task A kill rate` | Phase II 5 维**不可或缺**，确定性测试在 LLM 之前过滤了大量 bug |
| 重叠分析（A∩C / A∖C / C∖A） | 三类失败模式（"只有 Phase II + LLM 能杀" / "只有 LLM 能杀" / "都能杀"）的画像 |

| 单 mutant 平均耗时 | 1-2.5 分钟 |

## 6. Task B — Opus 改进 44 个 kernel

> **状态**：脚本尚未实现；prompt 是设计稿（见 [提示词全集 §5-6](提示词全集.md#task-b--strengthen_prompt_b设计稿)），等 Task A/C 跑完后再启动。

### 6.1 输入定义

| 字段 | 值 |
|------|----|

| 输出 | 强化后的 kernel.py 完整源码 + 改动 diff + V1/V2 验证报告 |

### 6.2 单 kernel 处理流程

```
对每个 kernel (共 44 个):
    1. 加载原 kernel.py（best_kernels.json）+ 参考实现（KernelBench）
    2. 调 evidence_collector.aggregate(kernel_name):
       返回 {phase1_*, phase2_*, taskA_*, taskC_*,
              survival_reason_clusters, top_unkilled_examples}

    attempt_history = []
    for round_num in 1..5:
        if round_num == 1:
            prompt = build_strengthen_prompt(
                kernel_code, reference_code, evidence)
            # 见 提示词全集 §5
        else:
            prompt = build_reanalyze_strengthen_prompt(
                kernel_code, reference_code, evidence,
                previous_attempts=attempt_history)
            # 见 提示词全集 §6

        保存 prompts/{kernel_name}_r{round_num}.txt
        llm_resp = call_opus_45(prompt)
        保存 llm_responses/{kernel_name}_r{round_num}_response.json

        parsed = parse_llm_response(llm_resp.content)
        if parsed is None or no strengthened_code:
            记录 parse_error; continue

        strengthened_code = parsed.strengthened_code
        # 写到 task_b_kernel_strengthen/{kernel_name}/strengthened_r{N}.py

        # === V1: 标准正确性 ===
        v1_result = verify_v1(
            problem_file, strengthened_code, n_seeds=5)
        # {passed: bool, failures: [{seed, max_abs_diff, first_diff_index}]}

        # === V2: 鲁棒性 ===
        v2_result = verify_v2(
            problem_file, strengthened_code, all 21 policies × 2 seeds)
        # {passed: bool, failures: [{policy, seed, max_abs_diff, diff_summary}]}

        attempt_history.append({
            "round": round_num,
            "hardening_strategy": parsed.hardening_strategy,
            "key_changes": parsed.key_changes,
            "strengthened_code": strengthened_code,
            "v1_result": v1_result,
            "v2_result": v2_result,
        })

        if v1_result.passed and v2_result.passed:
            success = True; break                            ← 双重通过

    保存 {kernel_name}/{evidence.json, prompts/, llm_responses/,
                       strengthened_kernel.py, validation.json,
                       diff_vs_original.txt}
```

### 6.3 双重验证（"V1 标准 + V2 增强"）的精确定义

| 验证 | 输入来源 | 通过条件 |
|------|---------|---------|
| **V1（标准正确性）** | KernelBench 参考实现的 `get_inputs()` × 5 个随机种子 | strengthened kernel 在每个种子上与 reference 输出 `bitwise-equal` 或 `allclose(atol=1e-2, rtol=1e-2)` |
| **V2（鲁棒性）** | 21 个 stress policy × 2 个种子（共 42 次差分） | strengthened kernel 在每个 (policy, seed) 上与 reference 输出对齐 |

**为什么不做"逆向变异分数"？**

逆向变异分数指：把改进后的 kernel 当新被测代码，再跑一次 mutation testing，看新的 mutation score 是否提升。
- 时间成本极高（44 kernel × 1646 mutant × stress 全套 ≈ 几十 GPU·小时）
- 改进的 kernel 可能产生新的等价类（让人难辨"分数提升来自更鲁棒"还是"等价类变化"）
- 用户/导师的判断：不在本次论文的核心 contribution 范围，省略

因此 Task B 只判定"改进是否破坏 V1/V2"——保证向下兼容。
