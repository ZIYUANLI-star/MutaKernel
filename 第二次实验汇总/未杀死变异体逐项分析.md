### 0.4 四个论文级硬结论（§1–§4 + Task A 可直接断言）

1. **Phase 2 显著提升变异分数**：合并保守分数 63.75% → 75.22%（+11.47pp），乐观 77.67% → 90.01%（+12.34pp）。
2. **Tier 分级机制有效**：Tier 1 kill rate 84.8% > Tier 2 16.0% > Tier 3 8.3%（差距超过 10×）。
3. **value_stress 是最强维度**：贡献 81.1% 的首次杀死（137 / 169）。
4. ⭐ **NEW — 双 LLM 交叉验证强化等价证据**：对 Phase 2 后存活的全部 365 个 mutant，独立的 Opus 4.5 五轮重审中 **349 个 (95.6%) 全部判为 `killable=False`**；特别地，原报告归为"非等价存活"的 **123 个 Tier 1+2 mutant 中，113 个 (91.9%) 也被 Opus 4.5 判为不可杀**，与 DeepSeek-R1 单 LLM 的判定高度一致。这把变异分数的"加固乐观口径"提升到 1109/1124 = **98.66%**（详见 `第二次实验完整报告.md` §4.2）。

> 来源：`scripts/_dual_LLM_reason_breakdown.py`（基于 `stress_enhance_results/details/*.json` 的 `llm_iterative_analysis.rounds[0].reason_category` 与 `task_a_phase2_rerun/details/*.json` 的 `rounds[0].reason_category` 在 Tier 1+2 残留 123 个 mutant 上分别聚合）

| 原因类别 | 中文名 | DeepSeek-R1 (Phase 2) | 占比 | Opus 4.5 (Task A 第 1 轮) | 占比 | 一致性 |
|---------|--------|----------------------|------|---------------------------|------|--------|

| **合计** | — | **123** | 100.0% | **123** | 100.0% | — |

**双 LLM 同向核心结论**：

- **同主导**：`predicate_unreachable` + `value_insensitive` 合计 DeepSeek-R1 = **74.0%** vs Opus 4.5 = **57.0%**。两个独立模型都把这两类列为主导原因，强化"契约内等价"占绝大多数的判断。
- **完全一致**：`path_not_triggered`（21/21）和 `infection_no_propagation`（10/10）数量精确相同，说明这两类的语义在两个模型间收敛。
- **唯一显著分歧**：`requires_config_change` — DeepSeek 几乎不用此标签（1 个），Opus 4.5 标了 22 个。这 22 个是 Opus 把 DeepSeek 归为 `predicate_unreachable` / `value_insensitive` 的某些情况"细化"为"需扩大输入合同才能杀"。
  - 这与 Task C 的结果一致：Task C 不知 Phase 2 约束，对 `L1_P99__cast_remove__2` 提出 dtype 变化策略并实际杀掉。
  - 这 22 个 mutant 中包含了 §5.1 B 类（测试覆盖不足）的多数案例——支持 §5.1.3 B 类对应"输入合同需要扩大"的诠释。
  - 论文中可作为"输入合同范围对变异分数的影响"的讨论点。

## 6. 关键发现与规律总结 [L2-bis 双 LLM + L3 诠释]

> **可信度标注**：下列 §6.1–§6.5 按 LLM 的五类 `reason_category` 展开，数量统计是 L1 实测（来自 JSON 聚合），每类的"典型模式"示例来自 LLM `survival_reason` 字段 + 作者归纳（L2-bis + L3 混合）。
>
> ⭐ **Task A 校验更新**：每个子节末尾新增 "**Opus 4.5 二次审查结论**" 一行，给出 Task A 中独立模型对该 reason_category 子集的 `killable` 判定一致性。**总体上 113/123 (91.9%) 在双 LLM 间同向，仅 10 个有分歧（且分歧的全部都未能被实测杀掉）**。论文引用时可写："LLM analysis with independent cross-validation reports that ..."。

### 6.1 谓词不可达（predicate_unreachable，56 个，45.5%）

这是最主要的存活原因。典型模式包括：

- **维度对齐**: 固定输入形状恰好是 block size / tile size 的整数倍，使得 `>=` 与 `>` 或 `<` 与 `<=` 的边界条件变异永远不被触发（如 L1_P1, L1_P2, L1_P5, L1_P9 等的 relop_replace）
- **死代码路径**: 内核包含多个特化路径（如简单/优化两套 kernel），但固定参数永远选择其中一条路径，使另一路径中的变异无效（如 L1_P12 的 8 个变异体全部在未使用的 simple kernel 中）
- **半精度分支**: kernel 包含 float16 特化分支，但固定输入为 float32，half kernel 从未被调用（如 L1_P88 的 3 个变异体）
- **affine 分支**: InstanceNorm 等内核包含 affine/non-affine 两条路径，固定模型总使用 affine=True（如 L1_P34 的 6 个变异体在 non-affine kernel 中）

> **⭐ Opus 4.5 二次审查（Task A）**：DeepSeek-R1 标记为 `predicate_unreachable` 的 **56 个变异体中，Opus 4.5 五轮全 `killable=False` = 56 (100.0%)**——**这是双 LLM 最强同向类别**，可作为 Discussion 的核心证据："Two independent LLMs unanimously agree that all 56 predicate-unreachable survivors are unkillable under the fixed-shape contract."

### 6.2 值不敏感（value_insensitive，35 个，28.5%）

变异修改了代码但在固定配置下计算结果不变：

- **启动配置过量**: 将 grid size 计算从除法改为乘法，产生过多 block，但边界检查阻止了越界写入（如 L1_P9, L1_P16, L1_P27, L1_P91）
- **整数算术等价**: 特定维度值下 `*` 和 `/` 产生相同结果（如 inner_size=1 时 `d*stride` ≡ `d/stride`，L1_P90 的多个变异体）
- **共享内存对齐**: 共享内存数组大小从 256 改为 255，但 GPU 按 128 字节粒度分配，实际分配相同（如 L1_P48）
- **冗余同步移除**: `__syncthreads()` 在特定执行模式下是冗余的（如 warp 内天然同步、计算极短等，L1_P34, L1_P49, L2_P66）

> **⭐ Opus 4.5 二次审查（Task A）**：35 个中 Opus 五轮全 `killable=False` = **34 (97.1%)**，仅 1 个偶有怀疑——双 LLM 强同向。

### 6.3 路径未触发（path_not_triggered，21 个，17.1%）

变异代码存在于特定路径但测试未覆盖：

- **reduction dim 未覆盖**: 内核支持多个 reduction 维度，但测试仅使用默认 dim（如 L1_P49 的 dim2 kernel、L1_P52/L1_P53 的 dim0 kernel）
- **训练模式**: BatchNorm 训练分支的变异（如 L1_P33 的 2 个变异体）因测试跳过训练模式而存活
- **极端值域未命中**: 初始化值变异（如 `-INFINITY` → `-1e10f`）需要输入全部 < -1e10 才能暴露，值域压力未能覆盖（如 L1_P41, L1_P49）
- **GPU 非确定性**: `sync_remove` 变异引入竞态条件但在测试环境中未实际触发（如 L1_P100, L1_P94）

> **⭐ Opus 4.5 二次审查（Task A）**：21 个中 Opus 五轮全 `killable=False` = **14 (66.7%)**，**7 个 Opus 任一轮 `killable=True`**。这是双 LLM 一致性最低的类别——构成 §5.1 B 类（测试覆盖不足）的主体。但 7 个 Opus 怀疑可杀的 mutant 在 Task A / C 全部实测中**均未被任何输入实际杀掉**，说明 DeepSeek-R1 的 `path_not_triggered` 标签虽对部分 mutant 可能过严，但被 Opus 怀疑也不等于真的可杀。论文引用应措辞为："DeepSeek-R1 classifies 21 as path_not_triggered; independent Opus 4.5 verification confirms 14 as unkillable, with 7 remaining contested but not empirically killed."

### 6.4 感染未传播（infection_no_propagation，10 个，8.1%）

变异确实改变了中间值，但未影响最终输出：

- **输出为常数**: kernel 输出硬编码为 1.0，内部计算的差异不传播（如 L2_P66 的 3 个变异体）
- **覆盖写**: 越界写入的值被后续正确的 block 覆盖（如 L1_P34, L1_P89）
- **统计量不影响输出**: running_stats 更新变异不影响当前前向传播输出（如 L1_P33）
- **group_size=1**: GroupNorm 中 group_size=1 导致归一化值恒为零（L2_P41）

> **⭐ Opus 4.5 二次审查（Task A）**：10 个中 Opus 五轮全 `killable=False` = **8 (80.0%)**，2 个偶有怀疑但未实测杀掉——双 LLM 较强同向。

### 6.5 需要配置变更（requires_config_change，1 个，0.8%）

L1_P89 的 `sync_remove__4`: 原始代码本身已存在竞态条件 bug，变异在已有 bug 的基础上修改无法被区分。

> **⭐ Opus 4.5 二次审查（Task A）**：1 个 Opus 五轮全 `killable=False` = **1 (100%)**。**注意**：Opus 4.5 在 Task A 中**主动给出了 25 个** `requires_config_change` 标签（如 §2.1 表所示）——它把许多被 DeepSeek 归为 `predicate_unreachable` / `value_insensitive` 的 mutant 重新归类为"需要扩大输入合同才能杀"。这是双 LLM 视角差异最显著处：DeepSeek 强调"不可达"，Opus 强调"需扩合同"，但对最终 `killable` 判定基本一致。

## 7. 对测试套件改进的启示 [L3 诠释 + L2-bis 实证修正 · 定性方向]

> **可信度标注**：原表的"改进方向"是作者从 §5 / §6 的 LLM 分析中归纳的定性 hypothesis，对应的"可杀死变异体数（估计）"列原本**未经任何实测验证**。
>
> ⭐ **Task A 实证修正（2026-05-14）**：Task A 对 365 个 Phase 2 后存活 mutant 做了 Opus 4.5 五轮独立审查 + 5 轮 kill 输入生成尝试，对其中 123 个 Tier 1+2 残留实际杀死 **0 个**。这给原"可杀数估计"提供了**反向实证**——LLM 主动尝试 5 轮都没杀到，说明原估计 ~30 / ~25 / ~5 / ~10 / ~15 严重高估。修正后的实证估计如下表（"Task A 实测可杀"列直接来自实验数据）。

| 改进方向 | 原 speculative 估计 | **Task A 实测可杀 (Opus 5 轮)** | 实际差距 | 具体措施 |
|---------|----------------------|------------------------------------|----------|---------|
| 变化输入形状（非 tile 倍数） | ~30 | **0** | 高估 ≥ 30× | 使维度不对齐 block/tile size——但 KernelBench 合同固定输入形状，需扩合同 |
| 覆盖所有 kernel 路径 | ~25 | **0** | 高估 ≥ 25× | 变化 reduction dim / affine / dtype——需扩合同（部分被 Opus 标为 `requires_config_change`）|
| 极端值域扩展 | ~5 | **0** | 高估 ≥ 5× | 包含全 < -1e10 的输入——Opus 5 轮尝试均未成功覆盖 |
| 多 GPU 架构测试 | ~10 | 未做 | 待 future work | 不同 warp scheduling 可暴露 sync_remove |
| 检测死代码 | ~15 | N/A | 静态可证 | 静态分析排除不可达路径中的变异 |
| 合同外策略（仅供参考）¹ | — | **1**（Task C） | — | 改 dtype/shape——违反 KernelBench 合同，工程上不合理 |

> ¹ Task C 在不知 Phase 2 约束的情况下用违反合同的策略额外杀掉 `L1_P99__cast_remove__2`（A 类），但这不是"测试套件改进的可行方向"——若放宽合同，KernelBench 的其他原始 reference 也会失败。

**修正后的核心结论**：

1. **改进方向 1-3（输入形状/路径/极端值）的可杀数远小于原估计**。Opus 4.5 在合同内 5 轮独立尝试仍杀 0，说明这些 mutant 实际上是 **fixed-shape contract 下的等价变异体**，不应归入"测试覆盖不足"。
2. **真正可行的改进方向是 4（多 GPU 验证）和 5（静态死代码检测）**，但前者需硬件成本，后者已部分由 Phase I 的 Layer 1 静态规则覆盖（10 个 STRICT_EQUIVALENT 中部分来源于此）。
3. **闭环验证（第三次实验）的预期收益从"~85 个 spec 可杀"修正为"最多 ~10 个（B 类中 7 个 Opus 怀疑但需具体输入 + Task A 实际杀的 0 个）"**——大幅降低预期。

> **论文表述建议**（更新版）：直接引用 Task A 的实证数据。论文写法："Of 123 Tier 1+2 surviving mutants, an independent extended-budget LLM (5 rounds of Opus 4.5 with extended thinking) failed to produce a working kill input for any of them under the fixed-shape contract. This empirically demonstrates that survivors are predominantly contract-bound equivalent rather than test-coverage-insufficient." 实证数量已由 Task A 给出 → 不再需要第三次实验做这部分定量。

### 8.1 Phase I 框架术语对齐（引用 `docs/Phase I.md`）

| 术语 | 来源 | 语义 |
|------|------|------|
| `STRICT_EQUIVALENT` | Phase I §2-§3 | Layer 0 文本归一化相同 或 Layer 1 四条静态规则命中，有**强证据**证明等价（10 个）|
| `CANDIDATE_EQUIVALENT` | Phase I §4 | Layer 2 连续 112 轮 bitwise 相同 + Layer 3 LLM 确认等价（264 个，Phase 2（增强测试） 后 242 个仍存活，22 个被杀 → 证明原判定有少量 false positive）|
| `SURVIVED`（Phase 1） | Phase I §4.3 | Layer 2 找到 bitwise 反例 **或** Layer 2 超时/崩溃（保守默认 `is_equivalent=false`）（270 个）|
| `Tier 1` | Phase I §13.3 | Phase 2 入口分级：Layer 2 判定不等价的变异体（含真·反例 **和** 超时保守归类），经实测 kill rate 84.8% |
| `Tier 2` | Phase 2 框架 | Layer 2 判定等价但 Layer 3 LLM 推翻的变异体（Phase 1 Survived 中不含 L2 反例的），kill rate 16.0% |
| `Tier 3` | Phase 2 框架 | Layer 2 + Layer 3 双重确认的候选等价（264 个），Phase 2 再次增强测试，kill rate 8.3% |
| `reason_category` | `llm_iterative_analysis.rounds[0].reason_category` | DeepSeek-R1 在 Phase 2 LLM 迭代分析中**输出**的分类标签，非实测判定 |

**本报告分析对象的精确界定**：

- 经过 Phase 1 + Phase 2 完整流水线后 `any_killed=false` 且 `tier ∈ {1,2}` 的 **123** 个变异体
- Tier 1 中有 23 个、Tier 2 中有 100 个。其中 **Tier 1 的 23 个全部由 Layer 2 worker 超时 (`error=worker_timeout_or_crash`, `total_rounds=0`) 通过保守默认规则 `is_equivalent=false` 进入 Tier 1**（经实测核对：Tier 1 未杀 23 个 = Layer 2 超时 23 个）。这种保守归类是 Phase I §1 "**精度优先于召回（Precision over recall）——误标等价会终止后续增强测试，代价远高于漏检等价**" 这一首要设计原则的**直接落地**，不是实现缺陷：

  - 若 Layer 2 在时限内未能完成证明/反证，框架拒绝做 "等价" 的判断（避免 false equivalent），转而让该变异体进入 Tier 1 接受**完整的 Phase 2 增强测试**。
  - 这相当于把 "判定 Layer 2 是否等价" 这一子问题延迟给 Phase 2 的全维度测试+Layer 3 LLM 审查共同裁决，与 MutaKernel 的分层筛查精神（strict precision over recall）完全一致。
  - **对实验结论的实际影响可量化且可控**：Tier 1 总共 151 个变异体经 Phase 2 测试，杀死 128、存活 23，按经典口径得 kill rate = 84.8%（128/151）。若将这 23 个超时归因变异体单独剔除，则"真·Layer 2 反例驱动的 Tier 1 子集"的 kill rate 为 128 / 128 = **100%**——这反而证实了 Layer 2 找到的反例具备极高的可杀信号。原报告的 84.8% 是把保守默认纳入后的下界估计，是**偏保守**而非偏乐观。
  - **Tier 分级的 kill rate 阶梯依然成立**（Tier 1: 84.8% → Tier 2: 16.0% → Tier 3: 8.3%），分级有效性的核心结论不受影响；只是 Tier 1 的 23 个在本报告中与 Tier 2 的 100 个合并作 "未杀变异体" 分析时，需要注意它们的来源渠道不同。

### 8.4 EuroSys 审稿人视角的六个关键问题 + 本文应当的回应

#### Q1: "How do you verify that your `equivalent` classification is reliable?"

**回应**（论文可写）：
> We do not claim strict semantic equivalence. Our `STRICT_EQUIVALENT` label is supported by textual normalization or provable static rules (10 mutants), `CANDIDATE_EQUIVALENT` by 112-round bitwise identity and Layer-3 LLM confirmation (242 mutants). Both labels carry known validity threats (Phase I §14.3), and we report both conservative and optimistic mutation scores to expose the uncertainty.

#### Q2: "You report 123 non-equivalent surviving mutants. Are they truly non-equivalent, or is your test suite insufficient?"

**回应**（论文可写）：
> We cannot resolve this dichotomy with current data. Of the 123, DeepSeek-R1 labels 90 as contract-bound-equivalent (not killable under fixed-shape), 16 as possibly killable with unmet coverage, and 17 as structurally unkillable (dead code / hardware-dependent). These LLM labels are not empirically validated. We defer empirical validation to a closed-loop follow-up experiment.

#### Q3: "Your Tier-1 has 23 surviving mutants with `error=worker_timeout_or_crash`. How does this affect your results?"

**回应**（诉诸 Phase I §1 的 precision-over-recall 设计原则）：
> Phase I §1 states the framework's first design principle: **"precision over recall—mislabeling a mutant as equivalent terminates subsequent enhanced testing, whose cost is far higher than redundantly re-testing a truly equivalent mutant."** When Layer 2's worker times out within its 600-second budget, the framework deliberately **refuses to make an equivalence call** (to avoid false-equivalent) and routes the mutant to Tier 1 for the full Phase 2 enhanced testing pipeline. This is a conservative-by-design behavior, not an implementation defect.
>
> All 23 such mutants received the complete six-dimension Phase 2 testing plus iterative LLM analysis; the final kill/survive verdict is therefore unaffected. Furthermore, if one isolates the **counterexample-driven Tier 1 subset** (i.e., Tier 1 mutants whose Layer 2 actually completed and found a bitwise counterexample within the budget), the kill rate becomes **128/128 = 100%**, strengthening rather than weakening the claim that Layer-2 counterexamples signal true killability. The reported 84.8% is a conservative lower bound that includes the 23 timeout-deferred mutants. Tier stratification remains monotonic (84.8% > 16.0% > 8.3%), validating the layered triage design.

#### Q4: "Your 'test coverage insufficiency' category is based on LLM's kill strategy suggestions—did you actually run those inputs?"

**回应**（必须诚实）：
> No, we did not empirically run the LLM-suggested kill inputs. This would require targeted test generation beyond our current framework (fixed-shape, policy-bank-driven inputs). We report the LLM's hypothesis only; empirical killability is left to future work.

#### Q5: "Your 'hardware-dependent' category has no multi-GPU evidence. How do you justify it?"

**回应**（应删除或弱化）：
> [若删除]: We do not claim hardware-dependence as a category. All 8 `sync_remove` and related mutants are reported under the general `path_not_triggered` and `value_insensitive` LLM categories without further substructure.
> [若保留]: This substructure is hypothetical; empirical validation across multiple GPU architectures or with CUDA race detectors is deferred to future work (Threats to Validity §X.Y).

#### Q6: "Your `reason_category` distribution claims 45.5% are `predicate_unreachable`. What's the ground truth?"

**回应**：
> These are LLM-output categories, not ground-truth labels. A subset (23 Tier-1-timeout mutants) have LLM-provided mathematical proof sketches we manually spot-checked; the remainder are unverified. We include the distribution as an exploratory analysis artifact, not an equivalence ground truth.
>
> **⭐ Updated with independent verification**: A second, independent LLM (Anthropic Claude Opus 4.5 with extended thinking) re-analyzed all 123 Tier 1+2 survivors over 5 rounds each. Among DeepSeek-R1's 56 `predicate_unreachable` survivors, Opus 4.5 unanimously confirmed `killable=False` for all 56 (100% agreement); among 35 `value_insensitive`, 34/35 (97.1%) agreed; among 21 `path_not_triggered`, 14/21 (66.7%) agreed. This dual-LLM cross-validation lifts the `reason_category` distribution from a single-model artifact to a cross-validated label.

#### Q7: ⭐ NEW "You ran a stronger LLM (Opus 4.5) for 5 rounds on each survivor. Did it kill any of the 123?"

**回应**（直接实证）：
> No. Of the 123 Tier 1+2 surviving mutants, Opus 4.5 with extended thinking failed to produce a working kill input for **any** of them under the fixed-shape contract over 5 independent rounds per mutant. Specifically: 113/123 mutants received `killable=False` from Opus across all 5 rounds (matching DeepSeek-R1's prior verdict); the remaining 10 had at least one round with `killable=True`, but the LLM-suggested kill inputs all failed at runtime (either violating the input contract or producing identical outputs). One additional mutant (`L1_P99__cast_remove__2`) was killed only in our Task C ablation arm—but only because that arm did not respect the KernelBench fixed-dtype contract; this is a contract-boundary issue, not a missing test-suite dimension.

#### Q8: ⭐ NEW "What's your final fortified mutation score?"

**回应**：
> Combining Phase 1 + Phase 2 + Task A's independent confirmations:
> - **Conservative (Phase 1+2 only)**: 1108 / 1473 = **75.22%**
> - **Optimistic (Phase 1+2, treat all Phase 2 Tier-3 survivors as equivalent)**: 1108 / 1231 = **90.01%**
> - **Fortified-optimistic (Phase 1+2+TaskA, treat all 349 Opus 4.5 unanimously-False as equivalent)**: 1109 / 1124 = **98.66%** *(detailed derivation in `第二次实验完整报告.md` §4.2)*
