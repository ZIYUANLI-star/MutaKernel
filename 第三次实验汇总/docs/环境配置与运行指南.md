### TritonBench-G 过滤说明（182 → 67）

TritonBench-G 注册表中共 182 个 Triton 语言 kernel，但其中 **115 个（63.2%）被自动过滤**，仅保留 67 个参与增强测试。原因如下：

**核心问题：缺少等价的 PyTorch 参考实现。**

MutaKernel 增强测试的工作原理是：对同一输入，分别运行 **PyTorch 参考实现**（reference）和 **被测 kernel**，比较两者输出是否一致。当输出不一致时记录为 discrepancy。这要求 reference 必须是与被测 kernel **语义等价**的正确实现。

TritonBench-G 中的许多 kernel 实现了特殊的 GPU 算法（如 Lightning Attention、Row-wise Dequantization 等），这些算法在 PyTorch 标准库中没有直接的等价函数。对于这 115 个 kernel，数据准备阶段只能生成 **恒等传递**（identity passthrough）作为 reference——即 `forward()` 直接返回输入本身，不做任何计算：

```python
# 有效的 reference（如 softmax kernel 的参考实现）
def forward(self, x):
    return torch.nn.functional.softmax(x, dim=-1)  # 真正计算 softmax

# 恒等传递 reference（如 lightning_attention kernel）
def forward(self, q, k, v):
    return q  # 直接返回输入，未做任何计算
```

**为什么恒等传递的 reference 会导致增强测试失效：**

增强测试的 5 个维度（value_stress、dtype_stress、training_stress、repeated_run、config_stress）均依赖以下判定逻辑：

```
original_ok = allclose(reference_output, kernel_output)
```

当 reference 是恒等传递时：
- `reference_output` = 原始输入（如 `q`）
- `kernel_output` = kernel 的真实计算结果（如 attention 计算结果）
- 两者**永远不相等** → `original_ok` 恒为 False → **每次都报 discrepancy**

这意味着所有 5 个维度构造的各种压力输入（极端值、不同数据类型、训练模式等）都无法有效检验 kernel 的正确性——因为 discrepancy 不是因为 kernel 在压力下出了 bug，而仅仅是因为 kernel 做了真实计算、结果不等于输入本身。**这些全是假阳性，不具有实验价值。**

剩余 67 个 kernel 拥有正确的 PyTorch 等价实现（如 softmax、relu、gelu、layernorm、matmul 等常见算子），可以正常进行增强测试。

## 四、运行实验

### 4.1 一键运行全部实验（推荐）

```bash
cd /path/to/MutaKernel
bash 第三次实验汇总/run_scripts/run_all_experiments.sh
```

脚本自动执行以下流程：

1. **环境检查** — 检测 Python、CUDA、GPU、依赖包
2. **数据完整性校验** — 验证所有 registry.json 和 problem 文件存在
3. **冒烟测试** — 每个数据集跑 2 个 kernel，验证环境可用
4. **全量测试** — 依次运行 apex → cuda-l1 → sakana → tritonbench

### 4.2 指定 GPU 架构

脚本默认会自动检测 GPU 架构。如果检测不准确，可以手动指定：

```bash
bash 第三次实验汇总/run_scripts/run_all_experiments.sh 8.6
```

常见 GPU 架构对照表：

| GPU 型号 | 计算能力 |
|----------|---------|
| H100 / H800 / H200 | 9.0 |
| **H20** | **9.0** |
| A100 / A800 | 8.0 |
| RTX 4090 / 4080 / 4070 | 8.9 |
| RTX 3090 / 3080 / 3070 | 8.6 |
| L40S / L4 | 8.9 |
| V100 | 7.0 |

### 4.3 仅冒烟测试

```bash
bash 第三次实验汇总/run_scripts/run_all_experiments.sh --smoke
```

### 4.4 单独跑某个数据集

```bash
cd /path/to/MutaKernel
export PYTHONPATH="$(pwd):$PYTHONPATH"
export TORCH_CUDA_ARCH_LIST="8.6"   # 按你的 GPU 设置

python3 scripts/run_fullscale_diff_test.py --dataset cuda-l1
python3 scripts/run_fullscale_diff_test.py --dataset sakana
python3 scripts/run_fullscale_diff_test.py --dataset tritonbench
python3 scripts/run_fullscale_diff_test.py --dataset apex
```

### 4.5 后台运行（防止 SSH 断连导致中断）

**推荐方式：tmux / screen**

```bash
tmux new -s exp
bash 第三次实验汇总/run_scripts/run_all_experiments.sh
# Ctrl+B, D 断开会话
# tmux attach -t exp 恢复
```

**备选方式：nohup**

```bash

tail -f experiment.log   # 查看进度
```

## 五、断点恢复

实验内置 checkpoint 机制。每完成一个 kernel，结果立即写入 `checkpoint.json`。

**如果实验中断（断电、OOM、手动停止），直接重新运行相同命令即可**，已完成的 kernel 会自动跳过。

```bash
# 直接重新运行，自动续跑
bash 第三次实验汇总/run_scripts/run_all_experiments.sh
```

## 八、保护机制

脚本已内置多层保护，防止 GPU 崩溃或进程卡死：

| 保护措施 | 说明 |
|----------|------|
| **显存限制** | 每个 worker 限制使用 90% GPU 显存，OOM 会抛异常而非冻结驱动 |
| **显存预检** | 每个 worker 启动前检查可用显存，不足 512MB 自动跳过 |
| **Worker 超时** | 单个测试用例最长 600 秒（10 分钟），超时自动 kill |
| **Kernel 总超时** | 单个 kernel 所有维度合计最长 3 小时 |
| **进程清理** | Kill 后自动清理子进程（包括编译器进程） |
| **Checkpoint** | 每完成一个 kernel 立即保存，中断后可续跑 |

## 九、常见问题

**Q: 编译报错 `nvcc fatal: Unsupported gpu architecture`**

设置正确的 GPU 架构（参见第 4.2 节对照表）：

```bash
export TORCH_CUDA_ARCH_LIST="你的GPU架构"
```

**Q: CUDA Out of Memory**

脚本会将 OOM 作为错误记录并跳过该测试用例，不会中断整体实验。如果频繁 OOM，建议使用显存更大的 GPU（>= 16GB）。

**Q: 实验中途卡住不动**

- 每个 worker 有 600 秒超时保护
- 每个 kernel 有 3 小时总超时
- 如果仍然卡住，`Ctrl+C` 终止后重新运行即可（checkpoint 自动保留已完成的 kernel）

**Q: 实验中断后如何恢复？**

直接重新运行同一命令即可，checkpoint 机制会自动跳过已完成的 kernel。

**Q: 想只重跑某个 kernel？**

删除 `results/{dataset}/details/{kernel_id}.json` 中对应的文件，并从 `checkpoint.json` 中删除该条目，然后重新运行。
