# 闭环验证实验方案：MutaKernel 增强测试作为生成循环 Validator

> **背景**: 导师提出需要补充一个闭环实验，证明 MutaKernel 的增强测试方法在实际 kernel 生成流程中的实用价值。
>
> **日期**: 2026-05-02

---

## 一、导师意图解读

### 1.1 原话还原

> "需要加个实验，让 AKO 或者其它 kernel 生成的工具用我们新开发的测试方法作为 validator 来跑。然后看看之前用 KernelBench 作为 validator 生成出来有 bug 的 kernel，用我们的 validator 跑生成出来的 kernel 是否是正确的。"

### 1.2 核心逻辑

这是一个**闭环验证实验** (Closed-Loop Validation)，目的是证明：

```
论证链:
  MutaKernel 发现了 validator 的弱点
    → 那把 MutaKernel 增强测试作为新的 validator
      → 放进 kernel 生成的迭代循环中
        → 之前因为 validator 弱而"通过"了的有 bug kernel，现在还能通过吗？
          → 如果不能通过，LLM 是否会被迫生成出真正正确的 kernel？
```

简单来说：**光说别人的 validator 弱还不够，要证明换上我们的 validator 后，生成出来的 kernel 确实变好了。**

### 1.3 论文中的位置

这个实验直接回应 `intro_review_notes.md` 中的两个关键审稿问题：

| 审稿问题 | 本实验如何回应 |
|---------|-------------|
| M2: 跨 benchmark 实验 / 实用性验证 | 直接展示 MutaKernel 作为 validator 替换后的生成质量提升 |
| P0: 需要证明增强方法的实际效果 | 从"分析型工具"升级为"可 plug-in 到生成 pipeline 的实用型工具" |

### 1.4 预期结果

| 对比场景 | KernelBench 原始 Validator | MutaKernel 增强 Validator |
|---------|--------------------------|--------------------------|
| Kernel X 生成结果 | ✅ 通过（实际有 bug） | ❌ 拒绝 → LLM 重新生成 → ✅ 真正正确 |
| Kernel Y 生成结果 | ✅ 通过（实际正确） | ✅ 通过（确认正确） |
| 总正确率 | 较低（假阳性高） | 较高（假阳性低） |

## 二、实验设计

### 2.1 实验总体架构

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        LLM Kernel 生成循环                              │
│                                                                         │
│  ┌──────────┐    ┌──────────────┐    ┌──────────────────┐              │
│  │  LLM     │───>│  生成 CUDA   │───>│    Validator     │──┐          │
│  │ (AKO等)  │<───│   Kernel     │    │  (可替换组件)     │  │          │
│  │          │    └──────────────┘    └──────────────────┘  │          │
│  │  修复/   │                                              │          │
│  │  重新生成│<──── ❌ 验证失败 ─────────────────────────────┘          │
│  └──────────┘                                                          │
│                        ✅ 验证通过 → 输出最终 kernel                    │
└─────────────────────────────────────────────────────────────────────────┘

实验变量:
  A组 (对照): Validator = KernelBench 原始验证 (固定输入 + allclose 1e-2)
  B组 (实验): Validator = MutaKernel 5 维度增强测试
```

### 2.2 两组 Validator 定义

#### A 组: KernelBench 原始 Validator (对照组)

```python
# KernelBench 默认验证逻辑 (来自 config.py)
atol = 1e-2
rtol = 1e-2
num_test_inputs = 5  # 固定形状、随机数值
# 逻辑: 5 次随机输入 → allclose → 全部通过则判定 correct
```

对应当前 `_stress_worker.py` 中 `policy_name = "__identity__"` + 默认 tolerance 的模式。

#### B 组: MutaKernel 增强 Validator (实验组)

使用 `run_fullscale_diff_test.py` 中实现的 5 维度测试：

| 维度 | 测试内容 | 检测能力 |
|------|---------|---------|
| `value_stress` | 21 种压力策略（极端值、边界值、稀疏值等） | 数值不稳定、溢出、精度丢失 |
| `dtype_stress` | float16 / bfloat16 降精度测试 | 低精度下的数值退化 |
| `training_stress` | .train() 模式 + 21 种压力策略 | 训练模式下的不一致行为 |
| `repeated_run` | 同一输入多次运行比较 | 非确定性 bug |
| `config_stress` | 多种 batch_size 测试 | shape-dependent bug |

判定标准：**所有 5 个维度均无 discrepancy 才判定为 PASS**。

### 2.3 被测 Task 选择

#### 来源 1: Sakana AI CUDA Engineer 中已知有 bug 的 kernel

从 AI CUDA Engineer Archive 中选取：
- **`Correct=True` 但被第三次实验检出 discrepancy 的 kernel**（假阳性案例，即 KernelBench validator 放过的有 bug kernel）
- 需要先完成第三次实验的全量测试，获取完整的 discrepancy 列表

#### 来源 2: KernelBench 原始实验中的 bug kernel

从第一/二次实验中：
- 在 KernelBench 默认 validator 下被判定 `Correct=True` 的 kernel
- 但在 Phase II stress testing 中被杀死的 survived mutant 对应的 task

#### 来源 3: CUDA-L1 中的 RL 优化 kernel

- RL 优化追求加速比，可能牺牲数值正确性
- 这些 kernel 在 KernelBench validator 下全部通过，但可能在增强测试下暴露问题

### 2.4 生成工具选择

| 工具 | 可行性 | 说明 |
|------|--------|------|
| **AKO (师弟)** | ✅ **首选** | 师弟已有的 LLM kernel 生成工具，需要确认 AKO 的 validator 接口约定 |
| **Sakana AI CUDA Engineer** | ✅ 可行 | 与 KernelBench 天然兼容，已有 Archive 数据可回溯 |
| **CUDA-L1 (RL 优化)** | ✅ 可行 | deepreinforce-ai 的 RL 生成流程 |

**协作模式**: 我们提供 `mutakernel_validator.py` 作为 drop-in 的 validator 替换组件，师弟在 AKO 的生成循环中调用即可（见第四节对接方案）。

## 三、实施方案

### 3.1 方案 A: 事后分析 (Post-hoc Analysis) —— 最快可行

> **核心思路**: 不需要重新跑生成循环，而是用已有数据做回溯分析。

#### 步骤

1. **从 AI CUDA Engineer Archive 筛选"KernelBench 通过但 MutaKernel 检出 bug"的 kernel**

```python
# 伪代码
for task in all_250_tasks:
    kernel = archive[task]["Correct==True 且 Speedup 最高的变体"]
    
    # A 组: KernelBench 原始 validator 判定
    kb_result = run_kernelbench_validator(kernel)  # 已知 Correct=True
    
    # B 组: MutaKernel 增强 validator 判定
    mk_result = run_5dim_stress_test(kernel)       # 第三次实验结果
    
    if kb_result == PASS and mk_result == FAIL:
        # 找到了! 这是一个 "假阳性" 案例
        buggy_tasks.append(task)
```

2. **统计核心指标**

| 指标 | 含义 |
|------|------|
| `N_total` | 总 task 数 |
| `N_kb_pass` | KernelBench validator 判定通过的数量 |
| `N_mk_fail` | 其中被 MutaKernel 增强测试检出 discrepancy 的数量 |
| `FPR = N_mk_fail / N_kb_pass` | KernelBench validator 的**假通过率** |

3. **对 buggy kernel 分析失败维度分布**

利用第三次实验 `details/*.json` 中已有的 `discrepant_dimensions` 字段，统计各维度的检出贡献。

#### 优点
- **无需重新跑生成循环**，用第三次实验的现有结果即可
- 最快出数据（第三次全量测试跑完即可分析）

#### 缺点
- 只能证明"MutaKernel 能检出更多 bug"，**不能直接证明"用了 MutaKernel 后 LLM 能生成出正确 kernel"**

### 3.2 方案 B: 在线生成循环 (Online Generation Loop) —— 完整闭环

> **核心思路**: 真正搭建一个 LLM kernel 生成循环，分别用两种 validator，对比最终生成质量。

#### 步骤

1. **搭建 kernel 生成循环脚本**

```python
def generate_kernel_with_validator(task, llm, validator_fn, max_rounds=10):
    """用指定的 validator 跑 kernel 生成循环。"""
    problem_file = get_problem_file(task)  # PyTorch reference
    
    for round_i in range(max_rounds):
        if round_i == 0:
            kernel_code = llm.generate(task.prompt)
        else:
            kernel_code = llm.repair(task.prompt, kernel_code, error_feedback)
        
        # 关键: validator_fn 是可替换的
        passed, error_feedback = validator_fn(problem_file, kernel_code)
        
        if passed:
            return kernel_code, round_i, "PASS"
    
    return kernel_code, max_rounds, "FAIL"
```

2. **定义两种 validator_fn**

```python
def kernelbench_validator(problem_file, kernel_code):
    """KernelBench 原始验证: 5次固定形状随机输入 + allclose(1e-2)"""
    for seed in range(5):
        cfg = {
            "mode": "value_stress",
            "problem_file": problem_file,
            "kernel_code": kernel_code,
            "mutated_code": kernel_code,  # 自身对比 ref
            "policy_name": "__identity__",
            "seed": seed,
            "atol": 1e-2, "rtol": 1e-2,
        }
        result = _run_stress_worker(cfg)
        if not result.get("original_ok"):
            return False, format_error(result)
    return True, ""


def mutakernel_validator(problem_file, kernel_code):
    """MutaKernel 增强验证: 5 维度全量测试"""
    result = run_kernel_5dim(problem_file, kernel_code, "gen_kernel")
    
    if result["total_discrepancies"] > 0:
        feedback = format_discrepancy_feedback(result)
        return False, feedback
    return True, ""
```

3. **对比实验矩阵**

```
对 N 个 task:
  A 组: generate_kernel_with_validator(task, llm, kernelbench_validator)
  B 组: generate_kernel_with_validator(task, llm, mutakernel_validator)
  
对比:
  - 最终 kernel 正确率 (用 MutaKernel 增强测试作为 ground truth)
  - 生成轮数 (迭代次数)
  - 有 bug 但"通过"了的 kernel 数量
```

4. **最终验证: 用 MutaKernel 增强测试统一评估两组结果**

```
A 组生成的 kernel → MutaKernel 5 维度测试 → 实际正确率 R_A
B 组生成的 kernel → MutaKernel 5 维度测试 → 实际正确率 R_B

预期: R_B >> R_A
```

#### 优点
- **完整闭环**，直接证明 MutaKernel 作为 validator 的实用价值
- 结果说服力最强，对 EuroSys 审稿人最具吸引力

#### 缺点
- 需要 LLM API 调用成本（但可以只选 20-50 个 task）
- 实现工作量较大（需要搭建生成循环）
- 5 维度测试耗时远超原始 validator（每个 kernel ~5-8 分钟 vs 原始 ~10 秒），会显著增加生成循环的总时间

### 3.3 方案 C: 半闭环混合方案 (推荐) —— 兼顾效率与说服力

> **核心思路**: 先用方案 A 筛出 buggy task，再只对这些 task 跑方案 B 的生成循环。

#### 步骤

1. **Phase 1: 事后筛选** (方案 A)
   - 完成第三次实验全量测试
   - 筛出 `Correct=True` 但有 discrepancy 的 kernel → `buggy_tasks`
   - 筛出 `Correct=True` 且无 discrepancy 的 kernel → `clean_tasks`

2. **Phase 2: 定向生成循环** (方案 B，仅对 buggy_tasks)
   - 对 `buggy_tasks` 中的每个 task，用 MutaKernel validator 重跑生成循环
   - 观察 LLM 是否能生成出真正正确的 kernel

3. **Phase 3: 汇总对比**

| 指标 | A 组 (KernelBench Validator) | B 组 (MutaKernel Validator) |
|------|----------------------------|----------------------------|
| 生成通过率 | 已知（来自 Archive） | Phase 2 结果 |
| 真实正确率 | Phase 1 筛出的比例 | Phase 2 用 MutaKernel 验证 |
| 假通过率 | `len(buggy_tasks) / N_kb_pass` | 预期为 0 |

#### 优点
- **效率最高**: 只对有 bug 的 task 跑生成循环，节省大量 LLM 和 GPU 时间
- **论证完整**: 既有全量统计数据（Phase 1），又有闭环生成验证（Phase 2）
- **实现可控**: 如果 buggy_tasks 只有 10-30 个，生成循环的成本完全可控

## 四、技术实现要点

### 4.1 给 AKO (师弟) 的对接方案

AKO 是师弟开发的 LLM kernel 生成工具。我们不需要改动 AKO 的代码，而是把 MutaKernel 的增强测试**封装为一个独立可调用的 validator**，让 AKO 在生成循环中替换原有的 KernelBench validator 即可。

#### 对接方式：命令行接口 (最通用，零耦合)

AKO 不管是什么语言、什么框架写的，都可以通过 subprocess 调用：

```bash
# AKO 只需在生成循环中调用这一行
python scripts/mutakernel_validator.py \
    --problem_file /path/to/problem_X.py \
    --kernel_code /path/to/generated_kernel.py \
    --mode full

# 返回 JSON 到 stdout，exit code: 0=通过, 1=有bug, 2=编译失败
```

输出示例：
```json
{
  "passed": false,
  "total_discrepancies": 5,
  "discrepant_dimensions": ["value_stress", "dtype_stress"],
  "feedback": "Kernel fails under extreme values (near_overflow: max_diff=1.23e+02) and low precision (float16: max_diff=3.45e+00). Fix numerical stability.",
  "details": {
    "value_stress": {"discrepancies": 3, "passes": 39, "worst_policy": "near_overflow"},
    "dtype_stress": {"discrepancies": 2, "passes": 4},
    "training_stress": {"discrepancies": 0, "passes": 42},
    "repeated_run": {"discrepancies": 0, "passes": 5},
    "config_stress": {"discrepancies": 0, "passes": 4}
  },
  "elapsed_s": 185.3
}
```

#### 对接方式二：Python API (如果 AKO 也是 Python)

```python
# AKO 侧只需 3 行代码即可接入
import sys; sys.path.insert(0, "/path/to/MutaKernel")
from scripts.mutakernel_validator import validate_kernel

result = validate_kernel(
    problem_file="/path/to/problem_X.py",   # PyTorch reference (含 Model + get_inputs)
    kernel_code="/path/to/generated.py",     # 生成的 CUDA kernel (含 ModelNew)
    mode="full",                             # "quick" | "standard" | "full"
)

if result["passed"]:
    print("Kernel 验证通过!")
else:
    # 把 feedback 传给 LLM，让它修复
    llm_feedback = result["feedback"]
```

### 4.3 AKO 侧需要满足的输入格式

AKO 生成的 kernel 需要符合 KernelBench 的标准格式（大概率已经满足）：

**problem_file (PyTorch reference)**:
```python
import torch
import torch.nn as nn

class Model(nn.Module):
    def __init__(self):
        super().__init__()
    def forward(self, x):
        return torch.nn.functional.softmax(x, dim=-1)

def get_inputs():
    return [torch.randn(16, 256)]

def get_init_inputs():
    return []
```

**kernel_code (生成的 CUDA kernel)**:
```python
import torch
from torch.utils.cpp_extension import load_inline

cuda_source = """..."""
softmax_ext = load_inline(name='softmax', cpp_sources=..., cuda_sources=...)

class ModelNew(nn.Module):
    def __init__(self):
        super().__init__()
    def forward(self, x):
        return softmax_ext.forward(x)
```

> **给师弟的一句话**: 只要你的 kernel 是 KernelBench 格式的（有 `Model` / `ModelNew` / `get_inputs`），直接调我的 validator 脚本就行，不需要改任何代码。

### 4.4 Error Feedback 格式 (给 LLM 的修复提示)

当 validator 判定失败时，`feedback` 字段会生成结构化的错误描述，AKO 可以直接传给 LLM：

```
Your kernel failed validation on the following test dimensions:
- value_stress: 3 discrepancies detected
  - Policy 'near_overflow': max_diff=1.23e+02, expected allclose(atol=1e-2)
  - Policy 'reduction_adversarial': max_diff=5.67e-01
  - Policy 'near_zero': max_diff=3.45e+00
- dtype_stress: 2 discrepancies in float16 mode

The kernel produces correct results with standard inputs but fails
under extreme value regimes. Please fix the numerical stability issues.
```

### 4.5 性能优化: 分层验证 (mode 参数)

5 维度全量测试耗时约 5-8 分钟/kernel。在生成循环中，AKO 可以按需选择验证强度：

```
mode="quick":    baseline (3 seeds, ~10s)     → 快速拦截编译错误
mode="standard": + value_stress (21策略, ~2min) → 拦截数值不稳定
mode="full":     + dtype/training/repeated/config (~5-8min) → 最严格

建议: 生成循环前几轮用 "quick"（快速迭代），最后一轮用 "full"（严格把关）
```
