# Diff-Test 编译缓存优化方案

## 问题描述

在全规模差异测试（`run_fullscale_diff_test.py`）中，每个 kernel 的同一份 CUDA 代码被重复编译最多 **8 次**，导致单 kernel 测试耗时从理论 1 分钟膨胀到 4-16 分钟。

### 根因分析

`_stress_worker.py` 对同一份 kernel 代码在 5 个测试维度中分别调用 `_load_module_from_source()`，每次使用不同的 `module_name`：

| 维度 | orig module_name | mut module_name |
|---|---|---|
| value_stress | `stress_orig_{HASH}` | `stress_mut_{HASH}` |
| training_stress | `train_orig_{HASH}` | `train_mut_{HASH}` |
| dtype_stress / repeated_run | `build_orig_{HASH}` | `build_mut_{HASH}` |
| config_stress | `cfgstress_orig_{HASH}` | `cfgstress_mut_{HASH}` |

`_patch_load_inline_names()` 将 `module_name` 作为后缀写入 `load_inline(name=...)`。PyTorch 的磁盘缓存以 extension name 为 key，不同后缀 → 不同 key → 每次重新编译。

在差异测试场景中 `mutated_code == kernel_code`（没有变异，只比较 orig vs ref），因此：
- 同一份代码被编译 **4 维度 × 2 (orig + mut) = 8 次**
- 每次 CUDA JIT 编译约 30-120 秒

### 时间消耗计算

| kernel 复杂度 | 单次编译 | 当前总编译 | 优化后 | 节省 |
|---|---|---|---|---|
| 简单（element-wise） | ~30s | 8×30s = 240s | 30s | 210s (87.5%) |
| 中等（矩阵运算） | ~60s | 8×60s = 480s | 60s | 420s (87.5%) |
| 复杂（fused attention） | ~120s | 8×120s = 960s | 120s | 840s (87.5%) |

## 优化方案

### 优化 1：Diff-Test 中跳过 mut 编译（省 4 次编译）

**原理**：差异测试中 `kernel_code == mutated_code`，编译 mut 并运行完全多余。

**修改文件**：`scripts/_stress_worker.py`

**修改位置**：`run_stress()` 函数中编译 mut 的部分

```python
# ========== 优化: diff-test 场景跳过 mut 编译 ==========
if cfg["kernel_code"] == cfg["mutated_code"]:
    # 差异测试模式: orig 和 mut 代码相同，无需重复编译
    mutant_ok = original_ok
    bitwise_orig_mut_eq = True
    mut_out = orig_out
else:
    # 正常变异测试模式: 需要编译和运行 mut
    try:
        mut_mod = _load_module_from_source(
            cfg["mutated_code"], f"stress_mut_{mut_hash}", tmp_dir,
        )
    except Exception as e:
        return {"ref_ok": True, "original_ok": original_ok, "mutant_ok": False,
                "ref_nan_fallback": ref_nan,
                "error": f"mut compile: {str(e)[:200]}",
                "time_ms": (time.time() - t0) * 1000}
    # ... 正常 mut 执行逻辑 ...
```

同样修改 `run_training_stress()`、`run_dtype_stress()`、`run_repeated()`、`run_config_stress()` 中对应的 mut 编译段。

**效果**：编译次数从 8 → 4。

---

### 优化 2：统一 module_name 前缀（省 3 次编译）

**原理**：让不同维度的 worker 使用相同的 extension name，使 PyTorch 磁盘缓存跨进程命中。

**修改文件**：`scripts/_stress_worker.py`

**修改方式**：将所有维度中的 module_name 统一为 `ext_orig_{HASH}` / `ext_mut_{HASH}`

```python
# Before (每个维度不同前缀):
# run_stress:     _load_module_from_source(code, f"stress_orig_{hash}", ...)
# run_training:   _load_module_from_source(code, f"train_orig_{hash}", ...)
# _build_models:  _load_module_from_source(code, f"build_orig_{hash}", ...)
# run_config:     _load_module_from_source(code, f"cfgstress_orig_{hash}", ...)

# After (统一前缀):
orig_mod = _load_module_from_source(cfg["kernel_code"], f"ext_orig_{orig_hash}", tmp_dir)
mut_mod = _load_module_from_source(cfg["mutated_code"], f"ext_mut_{mut_hash}", tmp_dir)
```

**效果**：第一个维度编译后，后续维度命中磁盘缓存。编译次数从 4 → 1（叠加优化 1）。

---

### 两者叠加效果

| 阶段 | 修改前 | 优化1 | 优化1+2 |
|---|---|---|---|
| 编译次数 | 8 | 4 | 1 |
| 编译耗时（60s/次） | 480s | 240s | 60s |
| 节省比例 | - | 50% | **87.5%** |

## 安全性分析：对变异测试无影响

### 变异测试中的保护机制

在变异测试场景下（`kernel_code ≠ mutated_code`）：

1. **Hash 天然不同**：`_code_hash(kernel_code)` → `HASH_A`，`_code_hash(mutated_code)` → `HASH_B`
   - 统一前缀后：`ext_orig_HASH_A` vs `ext_mut_HASH_B` → 仍然不同 → 不会冲突

2. **优化 1 的条件判断**：`if cfg["kernel_code"] == cfg["mutated_code"]` 在变异测试中为 `False`，走正常分支

3. **变异测试走不同代码路径**：第一次/第二次实验的变异测试主流程在 `MutantRunner.run_mutant()` 中，使用 mutant ID 作为 module_name，与 stress_worker 完全独立

### 极端情况

- 如果变异极小导致代码字面相同（理论上不可能，因为任何变异都会改变代码）：优化 1 的判断保证不会误判
- MD5 hash 碰撞概率：$2^{-80}$，实际不可能发生

## 附：进一步优化（可选）

### 优化 3：预编译 + 缓存复用

在 `run_kernel_5dim()` 入口处预编译一次 kernel，将编译好的 module 通过共享内存/pickle 传给各 worker。但这需要较大的架构改动（当前 worker 是独立子进程），且 torch extension 不支持序列化。不推荐在当前阶段实施。

### 优化 4：Worker 进程复用

将 5 个维度的测试合并到一个 worker 进程中顺序执行，避免跨进程缓存问题。但这会降低故障隔离性（一个维度的 GPU crash 会影响后续维度）。可作为二期优化考虑。
